# v0.0.1 Release Roadmap

## Summary

The project is in better shape than a prototype. The baseline build, `ctest`
suite, CI on Linux and macOS, and the optional test UI build all pass. The code
is also split cleanly across config loading, runtime lifecycle, API routing,
and session handling.

The right framing for `v0.0.1` is a local alpha release, not a hardened
self-hosted service. The current architecture fits a single-process, single
model server for localhost or a trusted LAN, with explicit operational limits.

There are no obvious architecture failures, but there are a few release
blockers to address before treating the project as a clean first tagged release:

- Empty `api_key` values silently disable auth instead of failing config
  validation.
- The README claims an MIT license, but the repository has no root `LICENSE`
  file.
- The README references `docs/zoo-keeper-metal-oom-issue.md`, but that document
  does not exist.
- "OpenAI-compatible" currently overstates the API surface. The server only
  accepts a narrow subset of chat completion fields, and server-owned tools are
  not actually installed yet.

## Recommended v0.0.1 Scope

Keep `v0.0.1` intentionally narrow:

- Single model
- Single process
- In-memory sessions only
- Localhost or trusted-LAN deployment
- No persistence
- No TLS termination
- No multi-tenant guarantees

Before tagging `v0.0.1`, complete these changes:

### Security

- Reject `api_key: ""` as invalid config.
- Emit a startup warning when binding a non-loopback address without API key
  auth enabled.

### API Contract And Docs

- Rename the public claim from "OpenAI-compatible" to "OpenAI-compatible
  subset".
- Document the exact request fields accepted by `POST /v1/chat/completions`.
- State explicitly that `/v1/tools` returns an empty catalog unless
  server-owned tools are added.

### Release Hygiene

- Add a root `LICENSE` file.
- Fix or remove the broken Metal OOM doc link in the README.
- Add short release notes that call out current limits.

### Operability

- Keep the current `/metrics` shape for `v0.0.1`.
- Document that request and error counters are based on HTTP response status and
  do not fully capture mid-stream failures yet.

## Roadmap After v0.0.1

### v0.0.2 Hardening

- Add live HTTP integration tests that boot Drogon with a fake `ChatService`.
- Exercise `/healthz`, auth rejection, `/v1/models`, `/v1/sessions`,
  stateless completions, and SSE cancellation paths end to end.
- Add request body size limits and clearer timeout behavior.
- Add explicit shutdown and signal-handling behavior.
- Improve streaming observability so partial stream failures are visible in
  metrics and logs.

### v0.1.x Performance, Usability, And Extensibility

#### Performance

- Add a benchmark harness.
- Publish tuning guidance for `context_size`, `n_gpu_layers`,
  `request_queue_capacity`, and history budgets.

#### Usability

- Add example configs for CPU-only, macOS/Metal, sessions-disabled, and
  API-key-enabled deployments.
- Add a documented error-code reference for the HTTP API.

#### Extensibility

- Replace the hardcoded startup-tools stub with a small server-owned tool
  provider interface.
- Do not add dynamic plugin loading yet.

Persistence and multi-model support should stay out of scope until the
tool-provider model and integration-test coverage are more mature.

## Public Interface Expectations

- `api_key` should be either `null` or a non-empty string.
- `POST /v1/chat/completions` should continue to behave as a documented subset
  until broader compatibility is intentionally implemented.
- Unsupported OpenAI request fields should keep returning `400 unknown_field`
  for now.
- `/metrics` should keep its current field names stable through `v0.0.1`.

## Test Plan

Keep the current baseline green:

```bash
cmake -S . -B build -DBUILD_TESTING=ON
cmake --build build --parallel
ctest --test-dir build --output-on-failure
```

Also keep the optional UI build green:

```bash
cmake -S . -B build-test-ui -DBUILD_TESTING=ON -DZKS_ENABLE_TEST_UI=ON
cmake --build build-test-ui --parallel
ctest --test-dir build-test-ui --output-on-failure
```

Add targeted tests for:

- Empty `api_key` rejection
- Non-loopback bind without auth warning
- README/license release hygiene
- At least one live HTTP integration path

Keep the real-model smoke test manual for now and document its prerequisites
instead of making CI depend on model artifacts.

## Assumptions

- `v0.0.1` is a fast local-alpha release.
- Sessions remain in-memory and process-lifetime only.
- Public internet exposure, TLS, production SLAs, and multi-tenant security are
  out of scope for this tag.
